# 勉強会の文字起こしから問いを抽出する

月1回の高次脳機能勉強会（2時間程度）の文字起こしを読み、**そこで立った問い**を
抽出して `content/questions/_inbox/` にスタブとして書き出す。

規約は `CLAUDE.md` の「高次脳機能部 wiki」節。**ここでの判断はすべてそれに従う。**

## この段階でやること／やらないこと

| | |
|---|---|
| やる | 文字起こしを読む / 問いを拾う / 暫定分類 / タイムスタンプの記録 |
| **やらない** | **調査（PubMed・Web 検索）／本文の執筆／採否の決定** |

調査と執筆は Phase 2。採否を決めるのは**人間（Toru）**であって、このスキルではない。
スタブを並べるところで必ず止まり、採否を尋ねること。

理由: 2時間の議論から出る「問い」の約半分は、活字にすると問いとして立っていない。
調査コストを払う前に人間が捨てられるようにするのがこの工程の目的。

## 入力

文字起こしのテキスト。**出所は問わない。**

| 形式 | 扱い |
|---|---|
| WebVTT (`.vtt`) | Zoom のクラウド記録。`00:01:23.456 --> ...` からタイムスタンプを取る |
| プレーンテキスト / Markdown | `[01:23:45]` や `1:23:45` 形式の時刻表記を拾う |
| tldv / Otter の書き出し | 話者名 + 時刻の行構造として読む |
| Zoom MCP | 接続が確立していれば `get_recording_resource` の `transcript`（`ts` が秒数で入る）|

**呼び出し例:**

```
/study-group transcripts/session-12.vtt 第12回 2026-07-15
```

引数が足りなければ聞くこと。必要なのは次の3つ。

- 文字起こしの場所（ファイルパス、または本文の貼り付け）
- 回数（`n`）
- 開催日

### タイムスタンプが取れない場合

規約4により、**原発言へ戻る導線は必須**。時刻表記が一切ない文字起こしを渡されたら、
勝手に `at: "0:00:00"` で埋めず、**その旨を報告して指示を仰ぐ**。
時刻なしで進める判断は人間がする。

## 何を「問い」として拾うか

拾うのは、**答えが自明でなく、調べるか論点を整理する価値があるもの**。

**拾う:**

- 明示的に問いの形で立ったもの（「これって結局〜なのか？」）
- 議論が紛糾して結論が出なかった論点
- 用語の使い方が人によって食い違っていた箇所（`type: hybrid` の候補）
- 「調べておきます」「宿題ですね」で終わった話題
- 臨床的な判断に関わるが、根拠が曖昧なまま流れた話

**拾わない:**

- 事実確認だけで済むもの（「あの薬の用量は？」）— ただし用語の定義が
  争点になっていれば `用語確認` として拾う
- その場で解決した質問
- 雑談・進行に関する発言
- 同じ論点の言い換え — **1つにまとめる**（重複スタブを作らない）

### 既存の問いとの重複を必ず確認する

規約6により、**同じ問いの再訪は新規ファイルを作らない。**

抽出後、`content/questions/*.mdx` の既存の問いと照合すること。同じ問いなら
スタブを作らず、**既存ファイルの `sessions` に1行足す提案**として報告する。

## 出力

`content/questions/_inbox/s{回数}-{slug}.mdx` に**frontmatter のみ**で書き出す。
本文は書かない（調査前に本文を書くと、根拠のない断定が混入する）。

```yaml
---
slug: semantic-memory-vs-understanding
title: 意味記憶の障害と「意味の理解」の喪失は同一か
type: map                 # 暫定。map | empirical | hybrid
domain: [semantic-memory, aphasia]
sessions:
  - n: 12
    at: "1:23:45"
quote: "……結局それって、覚えてないのと分かってないのって同じことなんですかね"
note: 用語の定義が話者間でずれたまま進んだ。概念整理が先に要る
---
```

| 項目 | 意味 |
|---|---|
| `slug` | 英小文字とハイフン。ファイル名と揃える |
| `title` | 問いの形にして書く。体言止めにしない |
| `type` | 暫定分類。判断根拠が薄ければ `note` に書いておく |
| `domain` | 既存の問いで使われている語を優先して再利用する |
| `sessions` | 立った回とタイムスタンプ |
| `quote` | **原発言の短い抜粋。** 採否判断の材料になるので必ず入れる |
| `note` | なぜ問いとして立つと判断したか。1〜2行 |

### `id` と `status` は書かない

- `id`（`q-NNN`）は**採用時に採番する**。未採用のまま番号を消費すると、
  捨てた問いの分だけ番号が飛ぶ
- `status` は書かない。**`_inbox/` にあること自体が「未採用」を表す**。
  `CLAUDE.md` の `status` は `open | mapped | evidenced` の3値で、
  ここに `inbox` を足すと wiki 側の状態遷移が濁る

`lib/questions.ts` は `content/questions/` 直下のファイルだけを読むので、
`_inbox/` の中身は wiki に出ない。

## 分類の指針

| type | 目安 |
|---|---|
| `empirical` | 前提を置けばエビデンスで詰められる。「〜は〜より予後が良いか」 |
| `map` | 原理的に解がない。「意識とは何か」「理解しているとはどういうことか」 |
| `hybrid` | 概念整理が先に要る。用語の定義が争点なら大半はこれ |

**迷ったら `hybrid` に寄せる。** `empirical` と誤分類すると Phase 2 で
概念整理を飛ばしてエビデンス探索に入ってしまい、噛み合わない調査結果が出る。

## 手順

1. 文字起こしを通読する。**長い場合も要約で済ませない**
   （問いは議論の細部に立つため、要約すると落ちる）
2. 候補を拾い、重複をまとめる
3. 既存の `content/questions/*.mdx` と照合する
4. `_inbox/` にスタブを書き出す
5. **一覧を提示して止まる。** 次の形で報告する:

```
第12回から 8 件の候補を抽出しました。

  1. s12-semantic-memory        [map]       意味記憶の障害と「意味の理解」の喪失は同一か
  2. s12-...                    [hybrid]    ...

既存の問いとの重複:
  - 「〜」は q-003 と同じ問いです。新規ではなく sessions への追記を提案します。

タイムスタンプを取れなかったもの: なし

どれを採用しますか。
```

## ゲート1（採用）

人間が採用を決めたら、次を行う。**この操作もスキルの担当範囲。**

1. `content/questions/*.mdx` の既存 `id` の最大値 + 1 を採番する
2. `_inbox/s12-foo.mdx` → `content/questions/q-0NN-foo.mdx` へ移動
3. frontmatter に `id` / `status: open` / `updated` を足し、`slug` `quote` `note` は
   本文に活かして落とす
4. `type` に応じた本文テンプレの見出しだけを置く（中身は Phase 2 で埋める）
5. 採用しなかったスタブは `_inbox/` に残す。**消さない**
   （次回以降に同じ問いが再燃したとき、既に検討した記録として効く）

本文テンプレの構成は `CLAUDE.md` の「本文テンプレは type で分ける」を参照。

## 注意

- **文字起こしをリポジトリにコミットしない。** 参加者の発言がそのまま入っており、
  このリポジトリは public。`quote` に載せるのは問いの成立に必要な最小限にとどめ、
  個人名・患者情報は含めない
- 抽出した問いに**答えを書かない**。この段階では問いの形だけを残す
- 文字起こしの読み取りに失敗した箇所（音声認識の誤りが疑われる等）は
  推測で補わず、`note` に「原発言の確認が必要」と書く
